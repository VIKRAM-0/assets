# Codebase Understanding — LIVINIT Fabric Configurator

Generated by reading every file in the repo. This is a **living reference**, not
marketing copy — it documents what the code actually does, file by file, including
the important functions/exports and how data flows between modules.

> **Note on `README.md` / `FEATURES.md`**: those two files describe an *earlier*
> architecture (a single 6,640-line `index.html`). The app has since been refactored
> (see `docs/superpowers/specs/2026-07-07-asset-designer-refactor-design.md`) into
> the modular `src/` structure documented below. Trust this document and the code
> over the top-level README for architecture questions.

## 1. What this app is

A single-page 3D furniture fabric configurator. Users:
1. Pick a product — **Sierra Chair**, **Haven Sofa**, or a **Bed** (wooden/fabric frame)
2. Apply real vendor fabrics (MityLite, Douglass, Ennis) to individual 3D mesh parts, by click or drag-and-drop
3. Tune PBR material sliders (brightness, roughness, metalness, sheen, pattern scale, bump strength)
4. Stage the piece in a 3D **room** (living room or bedroom), with a procedural **curtains/blinds** system
5. Search/analyze fabrics via **PolyHaven**, **AmbientCG**, and **Gemini AI** (photo → material properties)
6. Generate **AI photorealistic renders** (product hero shot, room shot, or "View in My Room" — composite onto the user's own room photo)
7. Export the configured model as a `.glb`

**Stack:** Vanilla JS + Three.js r128 (global `THREE`, loaded via CDN `<script>`, no bundler). ES modules in `src/` for app logic. Vercel static hosting + TypeScript serverless functions in `api/`. Tailwind Play CDN (preflight off) layered on top of a hand-rolled CSS design-system for a subset of components.

## 3. Architecture pattern: "classic scripts as ES modules"

Every file under `src/` is loaded as an ES module (`import`/`export`), but the code is
written in a **classic-script style**: functions and constants declared at module top
level, imported explicitly where needed. The key twist is `src/app/boot.js`, which
re-exports everything onto `window.*`:

```js
Object.assign(window, configurator, library, room, render, finder,
  { showPanelTab, toggleSidebar });
```

This exists because `index.html`'s markup still uses inline `onclick="..."` handlers
(e.g. `onclick="switchModel('chair')"`) — inline handlers execute in **global** scope,
so they can't see ES module bindings directly. The shim bridges that gap without
having to rewrite the markup's event wiring.

Two globals need special handling because they're **reassigned** (not just read) by
various modules:
- `window.appStore` — assigned once (stable singleton reference)
- `window.sph` — a `Object.defineProperty` **getter**, because `E.sph` is reassigned wholesale in several places (e.g. `E.sph = {theta:...}`); a one-time `window.sph = E.sph` would go stale after the first reassignment.

## 4. State management (`src/lib/`)

The codebase has **two parallel state mechanisms**, by design, mid-migration (see §7):

### 4.1 `store.js` — `appStore` (UI/app state)
A minimal pub/sub store:
```js
createStore(initialState) → { getState(), setState(updater), subscribe(fn) }
```
Holds: `sliders`, `baseColorHex`, `currentModelKey`, `roomMode`, `activeRoomSection`,
`curtainState`, `savedCurtainState`, `finder` (Fabric Finder modal state), `activeFabricKey`.

**Rule enforced by convention (not by code):** only functions in `actions.js` call
`appStore.setState`. Everything else reads via `appStore.getState()`.

### 4.2 `actions.js` — the only `setState` callers
`setSlider`, `resetSliders`, `setBaseColor`, `setModelKey`, `setRoomMode`,
`setRoomSectionState`, `setCurtain`, `saveCurtainState`, `restoreCurtainState`,
`setActiveFabric`, `setFinder`, `addCustomFabric` (pushes into `CUSTOM_FABRIC_ITEMS`,
which lives in `catalog.js` because 4 different `LIBRARY` groups hold a reference to
that same array).

### 4.3 `engine.js` — `E` (imperative Three.js/engine state)
**Deliberately NOT in the store** — the Three.js scene graph (meshes, materials,
textures, caches, camera) stays as a big mutable holder object `E`, because ES module
bindings can't be reassigned from other modules, but properties of an exported object
can (`E.sph = ...`). Also exports:
- `_gltfSceneCache` — pre-parsed GLB scenes keyed by URL (cloned per use)
- `roomFurnitureModels` — `{chair, sofa, bed_wooden, bed_fabric}` — kept alive simultaneously in room mode
- `polyCache` / `texCache` / `enhanceCache` — PolyHaven metadata / loaded-texture / AI-enhanced-texture caches
- `markDirty()` — sets `E._dirty = true`, read by the render loop (dirty-flag optimization: only re-render on state change)
- `CURTAIN_FABRICS` (9 fabric presets: linen, cotton, velvet, silk, voile, cotton-blend, wool, jacquard, blackout — each with roughness/opacity/normalScale/PolyHaven id/fallback textures/recommended colors)
- `CURTAIN_COLORS` / `CURTAIN_COLOR_GROUPS` — flat + grouped curtain color palettes
- `escapeHtml`, `showToast`, `setSliderVal`, `tryLoadTex` (cached texture loader), `makeGreyscaleTex` (canvas-filter greyscale conversion), `getPolyMaps` (PolyHaven files API + cache), `pickPolyUrl`, `saveMaterialSnapshot` (single implementation — was copy-pasted 5× pre-refactor)

### 4.4 `catalog.js` — static data
- GLB URLs: `CHAIR_GLB`, `SOFA_GLB`, `BED_WOODEN_GLB`, `BED_FABRIC_GLB`, `ROOM_GLB`, `BEDROOM_ROOM_GLB` (all routed through `/api/s3proxy?key=...`)
- `BEDROOM_SLOTS` — bed position/rotation/scale per bed type
- `POLY_IDS` — maps fabric `type`/`patKey` → PolyHaven asset id (e.g. `linen: 'rough_linen'`)
- `MATERIAL_MAPS` — per-type fallback normal/roughness texture URLs (Supabase/S3-proxied) used when PolyHaven is unavailable
- `CHAIR_WOOD`, `CHAIR_FABRICS` (MityLite Sierra — ~13 series, vinyl/fabric/PU/wood), `SOFA_KALEID`, `SOFA_TWIST` (Douglass #5308/#5881), `SOFA_LINUM`, `SOFA_CHALLENGER` (Ennis, hex-only, no images)
- `CUSTOM_FABRIC_ITEMS` — mutable array, the "My Fabrics" user collection, shared by reference across all model keys
- `LIBRARY` — the object actually rendered by `library.js`: `{ chair: [...groups], sofa: [...], bed_wooden: [...], bed_fabric: [...] }`, each group `{ group, vendor, vclass, items }`

## 5. Feature modules (`src/features/`)

### 5.1 `configurator/model.js` — GLB loading & mesh classification
- `buildMeshList()` / `buildPieceList()` — render the Parts checkbox list (product view) / Furniture Parts click-list (room view)
- `classifyMesh(mesh, modelKey, worldCenter, worldSize)` — shared naming logic: walks up the node-name tree against a `KNOWN` dictionary (`headboard→Frame`, `pillow→Pillow`, etc.); for beds, falls back to a volume/flatness heuristic (small+high = Pillow, flat+wide = Mattress, else Frame)
- `processGLTF(gltf)` — the main GLB→app pipeline: scales model to 1.6 world units, centers at origin, traverses every mesh/material slot, clones a neutral **greyMat** (color `#d4d0cc`, roughness 0.75, all texture maps stripped, double-sided) for fabric application, generates a greyscale version of any original diffuse texture, classifies each mesh (falls back to **positional heuristics** for chair/sofa: `Y<-0.55→Legs/Base`, `Z<-0.35→Backrest`, `|X|>0.55→Armrest`, `volRatio>0.12→Main Body`, `flatness<0.06→Stitching`), then applies model-specific renames (`RENAME` table) and restores any saved material snapshot for that model key
- `loadModel(url)` — loads a GLB (using `_gltfSceneCache` if already parsed) and calls `processGLTF`
- `switchModel(key)` — the chair/sofa/bed tab switcher; in room mode this does NOT call `processGLTF` (would re-center and clobber curtain state) — instead swaps which cached model is "active" and calls `_rebuildMeshEntries`
- `_rebuildMeshEntries(model, modelKey)` — rebuild `E.meshEntries` for a model already in the scene (room-mode tab switch), reusing `origMat._livinitGrey` if the material was already processed once
- `resetAll()` — deselects all parts, wipes all 4 model-key material snapshots, reloads the room companion model fresh

### 5.2 `configurator/materials.js` — the PBR material pipeline
This is the visual-quality core.
- `applySwatchToEntries(item, targetEntries)` — main entry point for both click-to-apply and drag-drop. Generation-token guarded (`_applyGen`, same pattern as `E._roomLoadGen`) so a stale in-flight apply can't clobber a newer one. Per-type defaults table:
  ```
  wood:    roughness 0.55  vinyl: 0.45  pu: 0.50  leather: 0.60  linen: 0.80  fabric: 0.72
  ```
  Texture resolution order: PolyHaven metadata (parallel-fetched) → `item._defaults` overrides → `MATERIAL_MAPS` fallback (Supabase). AmbientCG diffuse textures additionally run through `makeSeamlessTexture` + `enhanceTexture` (Gemini) before being applied. Physical repeat = `scale × (uvScaleFactor / BASE_TILE)` (`BASE_TILE = 0.3`) so tiling density stays consistent across differently-sized meshes. Special-cases bed models with smooth fabrics (leather/pu/vinyl): clamps `uvScaleFactor` tightly and lowers normal intensity to avoid patchy grain across UV islands.
- `makeSeamlessTexture(dataUrl)` — pure canvas algorithm, two steps: (1) de-light by blurring for a low-freq illumination field then dividing each pixel by it (flattens shading gradients), (2) shift-quadrant + smoothstep cross-fade blend so edges meet seamlessly at the tile boundary
- `enhanceTexture(dataUrl, cacheKey)` — POSTs to `/api/enhance-texture` (Gemini image-to-image), cached by `cacheKey`
- `handleDiffuseUpload(file)` — "replace with your own image" — swaps only the diffuse map, keeps existing normal/roughness maps, runs the seamless + enhance pipeline
- `initDragDrop()` / `startDrag()` / `dropFabricOnCanvas()` / `getHitEntry()` / `highlightHoveredMesh()` — raycaster-based drag-and-drop: casts a ray from camera through cursor, intersects `E.meshEntries` (+ curtain meshes in room mode), highlights hovered mesh with green emissive glow (`#2d4a3e`, intensity 0.35)
- `updateBrightness`, `applyProp` (roughness/metalness/sheen), `updateTexScale`, `updateNormScale` — live slider handlers, only affect checked/pieceSelected entries

### 5.3 `configurator/viewport.js` — Three.js scene & camera
- `initThree()` — creates renderer (ACES filmic tone mapping, sRGB output, physically-correct lighting), scene, camera, PMREM environment from `RoomEnvironment`, 4 lights (ambient + directional + hemisphere + fill), wires mouse orbit/pan/zoom (left-drag orbit, right-drag pan, scroll zoom clamped `[0.3, 30]`), a **floor-plane drag system** for furniture move-mode, and a **double-click** handler that enters/exits furniture move mode (custom HUD, not `TransformControls` gizmo). Contains the dirty-flag render loop:
  ```js
  (function loop(){ requestAnimationFrame(loop); if(!E._dirty)return; E._dirty=false; E.renderer.render(...); updateZoneLabelPositions(); })();
  ```
- `camUpdate()` — spherical→cartesian camera positioning from `E.sph {theta,phi,r}` and `E.tgt`
- `rebuildZoneOverlay()` / `updateZoneLabelPositions()` — floating clickable part-name labels projected from 3D mesh centers to 2D screen space, pushed outward from viewport center for readability
- `loadScripts(urls)` — parallel dynamic `<script>` loader for Three.js addons (DRACOLoader, GLTFLoader, RoomEnvironment, TransformControls)
- `handleGLBUpload(input)` — lets a user load their own `.glb`/`.gltf` file via `URL.createObjectURL`

### 5.4 `library/library.js` — fabric swatch UI
- `buildLibrary()` — renders the left-panel swatch grid from `LIBRARY[currentModelKey]`, flattens all groups into a searchable/filterable list, builds type-filter tabs (`fabric`, `vinyl`, `pu`, `wood`, `custom`, `all` — only tabs with items show), sticky per-series headers, "+ Add Fabric" button opens the Finder. Delegates to `buildCurtainLibrary()` instead when a curtain piece is selected in room mode.
- `renderActiveSwatch()` — syncs `.active` class from `appStore.getState().activeFabricKey` (single source of truth, replaced an old captured-DOM-element pattern)
- `buildCurtainLibrary()` — renders curtain fabric-type swatches + color chips instead of the normal fabric grid
- `_applyFabricFilters(typeKey, query)` / `filterFabricSearch(value)` — type + text search filtering with an empty-state message

### 5.5 `room/room.js` — room staging, curtains, blinds, placement (largest module)
- `toggleRoomView()` — the room mode on/off switch: saves a material snapshot, auto-selects living/bedroom section by current model type, calls `buildRoom()` or `buildBedroomRoom()`, sets a fixed interior camera angle. On exit, restores the product-view model transform and camera.
- `buildRoom(cb)` / `buildBedroomRoom(cb)` — load `ROOM_GLB`/`BEDROOM_ROOM_GLB`, scale to ~6 units, detect floor Y (largest flat XZ-footprint mesh), tag sub-objects for element toggles (walls/floor/windows/doors/rug/ceiling) by name substring match, detect curtain nodes and build curtain entries, place furniture. Both are generation-guarded (`E._roomLoadGen`) so a stale in-flight load can't stomp a newer one.
- **Curtain system**: `buildCurtainEntries(roomScene, opts)` is a unified builder shared by both room types (living room = single node `New6.002`; bedroom = two independent panels `curtains_1`/`curtains_2`). Captures base scale/bounding-box for the size sliders, applies a shared grey material, fixes mirror-flipped panel normals (`_fixCurtainNormals` — detects when one bedroom panel's baked normals point away from the room and flips them). `_buildCurtainMat()` assembles the actual material per shape (`sheer` → higher roughness + 42% opacity, `blinds` → matte, `pleated` → darker), using `MeshPhysicalMaterial` with a `sheen` Color for velvet's r128-era retroreflective rim. `_applyBlinds()`/`_buildBlindsGeometry()` — fully **procedural** venetian blinds (not GLB-sourced): tilted box-geometry slats + head/bottom rails + ladder cords, confined to a window-band fraction of the drape footprint (`SHADE_VCOVER=0.72`).
- **Furniture placement**: `FURNITURE_SLOTS` (chair/sofa x,z,rotY,scale for the living room) and `BEDROOM_SLOTS` (catalog.js) define calibrated positions. `_seatOnFloor(model, x, z, rotY, scale)` zeroes transform, applies slot scale on top of the 1.6-unit normalized scale, measures the post-scale bbox, then positions so the model's bottom sits on `roomFloorY`. `_placeFurnitureInRoom()` places the active model and (living room only) loads/positions the companion piece so both chair+sofa are visible together.
- **Move mode**: `toggleMoveMode()`/`setMoveMode()`/`nudgeFurniture()`/`rotateFurnitureY()` — double-click enters a custom HUD-driven move/rotate mode (floor-plane rayc税 drag), not the Three.js `TransformControls` gizmo (kept detached/invisible).
- **Explode view**: `updateExplode(val)` moves each mesh entry outward from model center along its own centroid direction; `animateExplode()` eases between 0↔1 over 1.2s (quadratic ease-in-out).
- `removeRoom()` — cleanup: restores curtain original materials, removes room group, keeps `roomFurnitureModels` cache alive for fast re-entry.

### 5.6 `render/render.js` — AI rendering & export
- `renderScene()` — main "Render" button handler. Positions a fixed hero camera angle (different for product vs room mode), captures 2 forced render passes (flushes the dirty-flag loop) to a JPEG data URL, HEAD-probes `/api/generate` (graceful degradation: if unavailable, just shows the raw screenshot), then POSTs to `/api/generate` with `mode: 'product'|'room'`.
- **"View in My Room"** (`openViewInMyRoom()` → `_vimrGenerate()`) — a **two-step AI pipeline**, distinct from the plain Render button:
  1. Capture the configured furniture (single asset or full living-room duo) on a clean neutral background via `captureSingleAssetScene()`/`captureDesignedScene()` (auto-framed to bounding box; curtains included if visible), POST to `/api/generate` with `mode:'furniture'` → get a clean photorealistic product shot
  2. POST that render + the user's uploaded room photo (+ optional curtain reference photo/text) to `/api/gemini-room` → get a composited "your furniture in your room" image
  Images are compressed client-side (`_vimrCompressImage`, canvas resize) to stay under Vercel's 4.5MB payload limit.
- `_vimrHideRoomExceptCurtains()` — hides room-GLB decor (walls/plants/bookcase) from the capture while keeping the user's configured curtains visible, so Gemini sees only the pieces relevant to the composite.
- `exportGLB()` — lazy-loads `GLTFExporter.js`, exports `E.currentModel` as binary GLB, triggers download as `{modelKey}_custom.glb`.

### 5.7 `finder/finder.js` — Fabric Finder modal (721 lines)
Two tabs, both writing into `appStore.getState().finder`:
- **Upload tab**: `handleFinderImage()` → `_analyzeImageAndAdd()` POSTs to `/api/find-fabric` (Gemini vision → JSON: name/type/roughness/sheen/metalness/scale/norm/hex/description), shown via `_showFinderAnalysis()`. User then either `saveAsMaterial()` (adds to `CUSTOM_FABRIC_ITEMS` without previewing) or `previewAnalyzedOnModel()` (runs it through `makeSeamlessTexture` then `applySwatchToEntries` immediately, shows the floating confirm bar).
- **Search tab**: `doFinderSearch()`/`quickSearch()` → `_searchAndShow()` queries PolyHaven (`_searchPolyHaven` — client-side keyword scoring against the cached `/assets?type=textures` list) and AmbientCG (`_searchAmbientCG` — via the `/api/acg-search` proxy) in parallel, merges/dedupes results, renders cards (`_renderFinderResults`). Clicking a card opens a preview drawer (`_showResultDrawer`) with two actions: `trySelectedOnModel()` (apply without saving) or `addSelectedResult()`/`_addFromSearchResult()` (fetch full diffuse/normal/roughness maps and commit to `CUSTOM_FABRIC_ITEMS`).
- **Floating confirm bar** (`showConfirmBar`/`hideConfirmBar`/`confirmAddFromBar`) — appears after a "Try on Model" preview so the user can commit it to "My Fabrics" without reopening the modal; reads `window._currentAppliedDiffUrl` so it captures any subsequent custom-image swap.
- **Custom image in search tab**: `handleSearchCustomImage`/`makeSearchImageSeamless`/`addSearchCustomToLibrary` — a mini upload flow embedded in the search tab, independent of the main upload tab.

### 5.8 `tour/tour.js` — onboarding
Self-contained IIFE (not a module with exports — wires `window._tour*` directly). 5-step spotlight tour (`STEPS` array with `target` CSS selector, `pos`, `title`, `desc`, `feats`). Persists "seen" state in `localStorage['livinit_tour_seen_v1']` so it only auto-shows on first visit. `_placeBubble()` computes bubble position/arrow direction relative to the spotlighted element, clamped to viewport.

### 5.9 `components/ui/panels.js`
Not a "feature" — a dedup utility. The Applied-Material-preview and Material-Adjustment-sliders blocks used to be duplicated markup (product panel + room panel). `SLIDER_ROWS` is a single spec array (brightness/roughness/metalness/sheen/scale/norm, each tagged `room:true` if it should also appear in the room panel with a `-r` id suffix); `sliderRowsHtml()`/`appliedPreviewHtml()` generate the HTML once and are inserted into the static markup shells at load time via `insertAdjacentHTML`.

## 6. Backend (`api/`, Vercel serverless, TypeScript)

All handlers use the Vercel Node function signature `(req, res) => ...` (no framework).

| File | Method | Purpose |
|---|---|---|
| `generate.ts` | POST | Gemini image generation. 3 prompt/model pairs keyed by `mode`: `product` (`gemini-2.5-flash-image` + `PRODUCT_PROMPT` — showroom hero shot), `room` (`gemini-3.1-flash-image-preview` + `ROOM_PROMPT` — strict-preservation interior photo), `furniture` (`FURNITURE_RENDER_PROMPT` — clean-background product photo, used as VIMR step 1) |
| `gemini-room.ts` | POST | "View in My Room" compositor — always `gemini-3.1-flash-image-preview`. `buildPrompt()` dynamically assembles a very detailed compositing prompt (shape-preservation rules + photorealism checklist covering lighting/shadows/perspective/surface interaction) from `singleAsset`/`assetLabel`/`curtainText`/`curtainImage` inputs. Takes 2–3 images (room photo, furniture render, optional curtain reference) |
| `find-fabric.ts` | POST | Gemini vision (`gemini-2.5-flash`) — analyzes an uploaded fabric photo, returns strict-schema JSON (name/type/roughness/sheen/metalness/scale/norm/hex/description) with defensive fallback defaults if the model's response isn't parseable |
| `enhance-texture.ts` | POST | Gemini image-to-image (`gemini-2.5-flash-image`) — cleans a diffuse texture: seamless edges, neutral lighting, sharper weave, exact color preservation |
| `acg-search.ts` | GET | Proxies AmbientCG v3 search API (browser can't call it directly — CORS); restricts to `Fabric,Leather,Carpet,Foam,AcousticFoam` categories |
| `acg-map.ts` | GET | Downloads an AmbientCG texture ZIP (via `jszip`) and extracts a single map file by regex pattern (`Color`/`NormalGL`/`Roughness`/`Displacement`/`AmbientOcclusion`), with a fuzzy fallback match |
| `s3proxy.ts` | GET | Streams an object from the S3 bucket at `fabric_assets/<key>` (query-param form: `?key=...`) |
| `s3-proxy/[...path].ts` | GET | Same as above but path-segment form (`/api/s3-proxy/a/b/c`), used by `viewer.html`; has verbose `console.log` debug tracing |

All GLB/texture URLs in `catalog.js` route through `s3proxy.ts` rather than hitting S3 directly (keeps AWS credentials server-side).

## 7. Config files

- **`package.json`** — deps: `@aws-sdk/client-s3`, `@google/genai`, `jszip`; devDeps: `@types/node`, `puppeteer-core` (used by `test/*.mjs`), `typescript`. No build script — this is intentional (see refactor doc, §8 below): "no bundler, no TS on the frontend, no framework."
- **`tsconfig.json`** — ES2022 target, CommonJS modules, `strict: false`, only compiles `**/*.ts` (i.e. `api/`).
- **`vercel.json`** — SPA rewrite (`/(.*) → /index.html`) + explicit `maxDuration` overrides for `acg-map.ts` (30s, ZIP download+extract) and `acg-search.ts` (15s).
- **`index.html`** — loads `styles/tokens.css` + `styles/app.css`, then the Tailwind Play CDN with `preflight: false` and a token-mapped theme extension (so Tailwind utility classes read the same CSS custom properties as the hand-written CSS — the redesign layers Tailwind `@apply` rules for a subset of components: `.seg-chip.active`, `.cp-section`, `.room-chip`, etc. — see the `<style type="text/tailwindcss">` block). Then Three.js r128 via cdnjs, then `<script type="module" src="src/app/boot.js">`.
- **`styles/tokens.css`** (122 lines) — Material-Design-flavored token system: space-separated RGB triples for alpha compositing (`--md-primary: 88 112 214`, used as `rgb(var(--md-primary) / 0.06)`), elevation shadows, ported from an external "web-pipeline lib/design-system".
- **`styles/app.css`** (633 lines) — the rest of the component CSS.

## 8. Tests (`test/`)

- **`smoke.mjs`** — headless Puppeteer (`puppeteer-core`, needs system Chrome — path is hardcoded to macOS `/Applications/Google Chrome.app/...`) end-to-end gate: boots the real app via `serve.mjs`, dismisses the tour, asserts the fabric library builds (≥50 swatches), the Finder opens/closes, a model loads (if the asset upstream is reachable — otherwise those checks are SKIPped, not failed), a swatch click executes, room view enters/exits (`appStore.roomMode` flips), and there are no uncaught page errors (tolerating known-flaky things: `/api/` 404s locally, CDN/texture-host network errors). Meant to be run before every refactor commit.
- **`serve.mjs`** — local static file server; proxies `/api/*` to the **production** Vercel deployment (`https://asset-designer-dev.vercel.app`) since serverless functions don't run locally without `vercel dev`. GET `/api/s3proxy` responses are disk-cached (`test/.asset-cache/`, sha1-keyed) so repeated test runs don't re-download GLBs, and fall back to hitting the S3 bucket directly if the Vercel deployment is paused/sleeping.
- **`perf-probe.mjs`** — Chrome DevTools metrics probe: measures at-rest style-recalc/layout churn (`page.metrics()` sampled twice, 3s apart, with zero interaction — should be ~0 if the dirty-flag loop is working), counts stuck-loading shimmer swatches, times a 30-frame scroll of the fabric grid.

## 10. Things worth knowing before changing code

- **Two state systems coexist on purpose.** `appStore` (UI/app state) is being migrated to piece-by-piece; the Three.js scene graph deliberately stays in `E` (imperative, outside the store) — don't try to put meshes/materials/textures into `appStore`.
- **Generation-token guards exist in three places** and follow the same pattern: `E._roomLoadGen` (room builds), `_applyGen` in `materials.js` (fabric applies), `_compGen`/local generation checks in async loaders. If you add a new async pipeline that can be superseded by a newer user action, follow this pattern (`const _gen = ++counter; ...; if (_gen !== counter) return;`) rather than trying to cancel in-flight promises.
- **`origMat._livinitGrey`** is a load-bearing side channel: once `processGLTF` clones a grey material for a mesh, it stashes the clone on the *original* material object so `_rebuildMeshEntries` (room-mode tab switches) can find and reuse the already-fabric-applied version instead of re-cloning from scratch.
- **Material snapshots** (`saveMaterialSnapshot`/`E.modelMaterialSnapshots`) are how fabric choices survive switching between chair/sofa/bed and entering/exiting room mode — always call `saveMaterialSnapshot()` after a material-mutating action if you're adding a new one, or state will silently not persist.
- **`CUSTOM_FABRIC_ITEMS` is a shared-by-reference array** across 4 different `LIBRARY` groups (chair/sofa/bed_wooden/bed_fabric all point at the same array for "My Fabrics") — never reassign it, only `.push()` (that's why `addCustomFabric` exists as the single mutation point).
- **All external asset URLs are proxied through `/api/s3proxy`**, not hit directly — this keeps AWS credentials server-side and lets `test/serve.mjs` disk-cache them.
